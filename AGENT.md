# CartridgeFlow Agent Rules

本文是后续 AI / Agent 进入本仓库后的强制执行入口。本文不替代 `TODO.md`、协议正文或实现代码；本文的作用是把协作纪律固定下来，防止再次出现无计划、大范围、不可回溯的修改。

## 0. 必读顺序

任何 Agent 开始工作前，必须按顺序阅读：

1. `TODO.md`
2. `docs/protocol/agent.md`
3. 与当前 TODO 编号直接相关的源文件

如果任务涉及协议升级，还必须阅读：

1. `skills/cartridgeflow-protocol-upgrader/SKILL.md`
2. `skills/cartridgeflow-protocol-upgrader/references/upgrade-checklist.md`
3. 当前协议正文和机器可读协议声明

## 1. TODO 是唯一变更入口

任何文件修改前，必须先确认本次修改对应的 TODO 编号。

允许直接做的事情：

- 只读检查。
- 查看 Git 状态。
- 阅读文件。
- 搜索代码。
- 向用户解释当前状态。

禁止直接做的事情：

- 没有 TODO 编号就编辑文件。
- 没有 TODO 编号就删除文件。
- 没有 TODO 编号就重构代码。
- 没有 TODO 编号就修改协议。
- 没有 TODO 编号就调整 UI。
- 没有 TODO 编号就重写卡带。

如果用户提出新任务，而 `TODO.md` 中没有对应条目，必须先更新 `TODO.md`，再执行实际修改。

## 2. 执行 TODO 的规则

执行任何 TODO 时必须遵守：

- 开始前把状态改为 `doing`。
- 执行过程中只修改该 TODO 允许的范围。
- 完成后更新状态为 `verify` 或 `done`。
- 完成后写明验收结果。
- 如果发现任务范围扩大，必须先回到 `TODO.md` 拆分或新增条目。
- 如果发现当前执行顺序不合理，必须先修改 `TODO.md` 的当前执行队列，并说明原因。

状态含义以 `TODO.md` 为准。

## 3. 协议文件规则

协议是项目边界，不是实现细节。

强制规则：

- 已发布协议版本只读保存。
- 不得为了修复某个卡带或某个 UI 问题直接改旧协议正文。
- 不得把开发底座的临时能力写成协议承诺。
- 不得在未实现、未测试、未声明基座能力前宣称支持某个协议版本。
- 协议语义变化必须创建新版本。
- 新版本必须同步文档、机器可读声明、能力词表、基座声明和测试。

当前协议入口是 `docs/protocol/agent.md`。当前优先协议是 `CF-FARP@0.4`。

## 4. 修改代码的规则

修改代码时必须保持收敛：

- 优先读现有实现，再决定改法。
- 优先最小修改，不做无关重构。
- 不批量改 UI 结构。
- 不把多个问题塞进一次大改。
- 不回滚用户改动，除非用户明确要求。
- 不使用破坏性 Git 命令。
- 删除目录前必须确认绝对路径在当前工作区内。
- 涉及前端 UI 时，先审计对应组件和样式，再改一个明确问题。

任何大范围改动都必须拆成多个 TODO。

## 5. Git 规则

提交前必须：

- 确认 `git status`。
- 说明本次提交包含哪些 TODO 编号。
- 确认没有把运行产物、缓存、依赖目录、日志、大体积临时文件提交进仓库。
- 确认没有提交 API Key、账号凭证或生产密钥。

建议：

- 一项 TODO 一个提交。
- 重大修复先提交检查点，再继续下一项。
- 发布标签不得复用旧含义。

## 6. 当前工作焦点

当前项目正在收敛，不再继续堆叠无计划功能。

优先级顺序以 `TODO.md` 的“当前执行队列”为准。当前方向包括：

- 恢复变更纪律。
- 建立工程状态快照。
- 明确可恢复基线。
- 审计测试台 UI。
- 修复日志栏、详情面板、画布交互。
- 梳理 AI 决策真实闭环和 `decision_envelope` 消费链路。
- 核对 `CF-FARP@0.4` 支持情况。
- 重建符合协议的 `dev.pixel_episode_director` 卡带流程。

## 7. 交付标准

每次完成任务后，最终回复必须说明：

- 执行了哪个 TODO 编号。
- 修改了哪些文件。
- 验证了什么。
- 哪些风险或待办仍然存在。

如果没有运行测试，必须明确说明没有运行。
