# CartridgeFlow 系列视频生产目标

更新日期：2026-07-18

## 最终目标

利用用户自己收集并确认可用的角色、场景、道具和动作素材，让用户输入一段短文本后，由 CartridgeFlow 协调 AI、Blender、ComfyUI 和人工复核，在当前电脑上稳定生产可发布的系列短视频。

第一阶段不追求“任意文本一键成片”，而是建立一条受控、可重复、可返工的镜头级半自动生产线。

```text
短文本
-> 结构化剧本与分镜
-> 固定资产和动作匹配
-> Blender 场景、动作、镜头和控制素材
-> ComfyUI 单镜头增强
-> 人工复核与局部返工
-> 配音、字幕、剪辑和 MP4 交付
```

## 当前硬件边界

- GPU：NVIDIA GeForce RTX 3080，12288 MiB 显存。
- 内存：约 32 GB。
- 当前 PowerShell PATH 尚未发现 Blender、FFmpeg 和 Python；这不等于未安装，第一步必须确认实际安装位置和可调用方式。
- ComfyUI 必须以本机实测为准，不凭模型宣传决定生产方案。

3080 12GB 的生产原则：

- 一次只处理一个镜头或一个任务，禁止并行抢占显存。
- 先做短镜头、低分辨率控制素材，再合成或放大到交付分辨率。
- 优先使用参考图、控制视频、图生视频或视频到视频，不依赖长段纯文本视频生成。
- 固定并记录 workflow、模型版本、seed、分辨率、帧数、耗时和峰值显存。
- 显存不足或工作流不可用时明确失败，不静默换成假结果。

## 四层职责

### 用户素材是资产源

用户负责选择最终采用的角色、场景、道具、动作和声音素材。系统负责登记路径、版本、许可来源、缩略图、标签以及兼容性，不擅自用 AI 替换正式素材。

### Blender 是空间事实层

Blender 负责真实资产装载、骨骼动作、角色走位、场景结构、摄像机、灯光和时间轴，并输出原始预演、参考帧、深度、法线、遮罩或控制视频。远景和复杂空间关系以 3D 结果为准。

### ComfyUI 是画面增强层

ComfyUI 只负责经过选择的镜头类型，例如风格化、细节增强、局部重绘、短动作迁移或视频到视频。它不负责保存世界状态，也不替代资产、分镜和镜头调度。

### CartridgeFlow 是生产调度层

CartridgeFlow 负责结构化决策、工具调用、用户确认、运行暂停、产物登记、质量门禁、失败经验和局部重跑。通用能力放在底座，系列视频规则留在卡带。

## 第一条真实样片

第一条样片必须主动限制范围：

- 1 个固定主角。
- 1 个固定场景。
- 最多 1 个关键道具。
- 3 个已经可用的动作。
- 3 至 5 个镜头。
- 总时长 15 至 30 秒。
- 先用旁白或无对白，暂不做复杂口型。
- 竖屏 1080x1920 MP4 为最终交付规格，但中间生成分辨率由显存实测决定。
- 只接入 1 个经过基准测试的 ComfyUI 工作流。

## 完成定义

只有同时满足以下条件，才算证明这条生产路线成立：

1. 连续完成 3 集同角色、同场景的短片，每集 15 至 30 秒。
2. 每集可从短文本生成结构化剧本、分镜和资产匹配结果。
3. Blender 使用真实模型和动作生成可播放预演，不再以方块代理作为交付结果。
4. 至少一种镜头可以在 RTX 3080 12GB 上由 ComfyUI 稳定增强。
5. 用户可以在剧本、分镜和成片复核点确认或驳回。
6. 驳回单个镜头时只重跑相关链路，并携带上一次失败原因。
7. 每次交付包含 MP4、镜头清单、资产清单、工作流参数和生产报告。
8. 三集完成后，能明确统计人工耗时、GPU 耗时、失败率和可复用镜头比例。

## 实施阶段

### M0：本机生产基线

确认 Blender、FFmpeg、Python、ComfyUI、模型和 API 的真实位置与版本。选择候选工作流，在 3080 上记录分辨率、帧数、耗时、峰值显存和输出质量，最后只保留一个首发工作流。

验收产物：`docs/production/ENVIRONMENT_BASELINE.md` 和一组可复现的 ComfyUI 基准结果。

### M1：真实 3D 单镜头

将一个真实角色、一个场景和一个动作登记进资产库。替换当前硬编码代理生成逻辑，让卡带真正调用 Blender 装载资产、绑定动作、设置机位并输出一个可播放镜头。

验收产物：真实资产生成的 Blender 工程、预演 MP4、镜头 manifest；该阶段不依赖 ComfyUI。

### M2：单镜头 ComfyUI 增强

把 M1 镜头整理成 ComfyUI 可消费的控制包，通过本地 API 提交唯一首发工作流，轮询状态并回收结果。保留 3D 原片作为对照和失败回退来源。

验收产物：原片、增强片、workflow JSON、参数 manifest、显存与耗时记录。

### M3：第一条 15 至 30 秒样片

扩展到 3 至 5 个镜头，打通文本、剧本、分镜、资产匹配、Blender、ComfyUI、复核、局部返工、声音和合成交付。

验收产物：第一条可以公开观看的竖屏 MP4，以及完整生产报告。

### M4：三集复现

使用同一角色和场景连续生产三集，只改变文本、表演组合和镜头编排。根据真实失败数据修正工作流，不在这一阶段扩展新题材。

验收产物：三集成片、失败分类、平均生产时间和下一阶段决策。

## 决策门

- M0 未完成前，不决定采用哪种视频模型，也不开发多个 ComfyUI 接口。
- M1 未完成前，不继续扩展代理预演的视觉效果。
- M2 单镜头不能稳定复现时，不进入整片增强。
- M3 未形成真实 MP4 前，不做发布平台、数据回流和大规模模板库。
- 三集复现未通过前，不增加多人对白、复杂口型、长视频或模型训练。

## 明确不做

- 不让 AI 自动生成正式角色、场景、道具和动作资产。
- 不以任意题材、任意角色的一键成片为近期验收目标。
- 不同时接入多套视频模型和工作流。
- 不为了单个卡带修改已发布协议语义。
- 不继续做与真实镜头生产无关的 UI 美化或通用重构。
- 不把静态代理、Mock 输出或静默降级结果当作真实生产能力。

## 当前行动准则

后续每项工作都必须回答一个问题：它是否直接帮助当前里程碑产生真实、可复现、可复核的镜头产物？

如果答案是否定的，就不进入当前 TODO。
