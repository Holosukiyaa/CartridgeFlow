# CartridgeFlow TODO

文档版本：2026-07-17.1  
当前原则：任何修改必须先进入本文件，再按编号执行。  
适用范围：协议、基座、后端、前端、测试台、设计台、卡带、文档、Git 发布流程。

## 0. 强制规则

### 0.1 TODO 是唯一变更入口

从本版本开始，所有修改必须满足以下条件：

- 修改前必须在本文件登记 TODO 条目。
- 执行时必须引用 TODO 编号。
- 完成后必须更新 TODO 状态和验收结果。
- 没有 TODO 编号的改动只能做只读检查，不得编辑文件。
- 临时修复、样式微调、协议补丁、卡带重写、清理文件、提交发布都不例外。

### 0.2 协议文件不可被随意覆盖

- 已发布协议版本只读保存。
- 发现协议不足时，先在 TODO 中登记“协议变更提案”。
- 只有用户确认后，才能创建新协议版本。
- 不得为了让某个流程通过，直接改旧协议正文。
- 不得把开发底座的临时能力写成协议承诺。

### 0.3 代码修改必须最小化

- 不做无关重构。
- 不批量改 UI 结构。
- 不用一次大改覆盖多个问题。
- 不回滚用户改动，除非用户明确要求。
- 修改前必须先确认受影响文件和现有实现。
- 修改后必须给出验证结果。

### 0.4 Git 管理规则

- 每次提交前必须说明提交包含哪些 TODO 编号。
- 提交前必须确认 `git status`。
- 不把运行产物、缓存、依赖目录、日志、大体积临时文件提交进仓库。
- 不把 API Key、账号凭证、生产密钥提交进仓库。
- 重大修改建议一项任务一个提交。

## 1. 状态定义

- `todo`：尚未开始。
- `doing`：正在执行。
- `blocked`：被外部条件或用户决策阻塞。
- `verify`：实现完成，等待验证。
- `done`：已完成并通过验收。
- `dropped`：经确认不再执行。
