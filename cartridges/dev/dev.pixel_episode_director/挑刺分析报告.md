## 0. 一句话结论

这条"像素短剧工厂"流水线**画得很漂亮，但一大半是仪表盘装饰**：8 个 AI 调度节点不影响任何结果、多个关键数据键从未被生产、资产门禁在"每次强制重造资产"之后才检查所以形同虚设、被声明为 approved 的角色贴图文件在仓库里根本不存在。三份说明文档描述的流程结构与真实 `root.flow.json` 严重不符。MCP 工具库右侧栏无法深入配置，是因为 `params_schema` 字段全空且前端根本不消费它。

> **更正（第三轮）**：ComfyUI 升级链路的"孤立"是**作者刻意为之的隔离特性**，不是缺陷（见第 3 条）。真正的问题不在"它被隔离了"，而在"**测试台分不清故意隔离和意外断链**"——这正是本轮要修的核心：**测试台太弱，我在前两轮查出的所有严重 bug 全都能在测试台"通过"的情况下溜过去。** 详见第 15 条。

严重程度分级：🔴 致命（流程跑不出正确结果） / 🟠 严重（设计自欺 / 浪费） / 🟡 一般（体验 / 维护性）。

## 1. 🔴 数据链断裂：多个 input 键从未被生产

这是最致命的问题。逐节点追踪 store 键的生产与消费：

| 节点 | 声明的 `input` | 该键的生产者 | 实际状态 |
|---|---|---|---|
| `dispatch_material_reads` | `production_recipe` | 无 | ❌ 从未生产 |
| `plan_asset_check` | `production_context` | 无 | ❌ 从未生产 |
| `plan_draft_asset_request` | `production_context,asset_check_report` | `production_context` 无 | ❌ 部分缺失 |
| `assemble_director_package` | `production_context,asset_check_report,draft_asset_report` | `production_context` 无 | ❌ 合并包缺原料 |

**证据**：`collect_episode_brief`（root.flow.json:54-83）的输出是 `episode_brief`。而说明文档里负责产出 `production_recipe` 的 `store_recipe_sheet` 节点、负责产出 `production_context` 的 `merge_production_context` 节点，**在 `root.flow.json` 里根本不存在**（states 里搜不到这两个 ID）。

**后果**：
- `_read_input`（node_executor.py:208-219）只会拼接 store 里**已存在**的键，缺失键被静默跳过。所以这些 AI 节点拿到的是**空上下文**或残缺上下文。
- `assemble_director_package` 组装出的 `director_package` 里根本没有用户填的产品配方/剧情目标——导演包丢失了最核心的原料。
- 之所以流程还能"跑通"，是因为真正干活的工具节点（如 `generate_shot_plan`）绕开了这套 package 抽象，直接用 `store:episode_brief.episode_goal` 等硬编码引用取值（root.flow.json:552-559）。**也就是说整套"传递节点 / 物料总线"抽象是空转的。**

## 2. 🟠 8 个 AI "调度"节点是纯装饰，不影响任何下游结果

流程里每个工具节点前面都挂了一个 `llm_prompt` 节点（`dispatch_material_reads`、`plan_asset_check`、`plan_draft_asset_request`、`plan_shot_plan_request`、`plan_validate_request`、`plan_render_request`、`plan_mux_request`、`plan_world_state_update`），号称"调度 AI / 质检 AI / 导演 AI"。

**证据**：这些节点的 prompt 全是固定一句话，如"请调度四台资料入库机器"（root.flow.json:107）、"请把导演包转成 shot_plan.json 生产指令"（root.flow.json:499）。它们的输出写到 `material_read_orders`、`shot_plan_request` 等键。

**但下游工具节点根本不读这些输出的语义**：
- `load_series_bible` 的 `input` 虽写 `material_read_orders`，但它的工具参数是硬编码 `path: assets/series_bible.json`（root.flow.json:151-158）。AI 说什么都不影响读哪个文件。
- `generate_shot_plan` 的 `input` 写 `shot_plan_request`，但工具参数全是 `store:episode_brief.*`（root.flow.json:551-560），跟"导演 AI 的生产指令"无关。
- `asset_check` 的 `input` 写 `asset_check_request`，但 `check_pixel_assets` 用的是硬编码路径（root.flow.json:359-363）。

**后果**：
- 每跑一次卡带，白白发起 **8 次 LLM 调用**，纯烧 token/时延，对产物零贡献。
- 违背了文档自称的"AI 节点是调度员和质检员"——它们既不调度也不质检，只是往 store 里塞了 8 段没人看的话。
- 更糟：若 LLM 未配置，`_offline_llm_response`（node_executor.py:360-419）会生成一份**文件总结体裁**的 Markdown（它是从 `dev.file_summary` 抄来的兜底），跟镜头规划毫不相干，进一步坐实这些输出无意义。

**建议**：要么让工具真正消费 AI 输出（把 shot plan 的剧情结构交给 LLM 生成，而不是 `_build_pixel_shot_plan` 里的确定性模板），要么直接删掉这 8 个空转节点，省 8 次调用。当前状态是"两头不占"。

## 3. ✅ ComfyUI 升级链路是「故意隔离的孤立分支」——这是特性，不是 bug（已更正）

> **本条为第三轮更正**。第一、二轮曾把 `remote_comfy_upgrade` 定性为"死代码"，这是**误判**。经作者澄清：升级通道是**当前刻意隔离出主链路**的，因为现阶段专注于非升级链路的调整。能被静态分析一眼看出"这个节点游离在主链路之外"，恰恰**证明这套流程的模块隔离做得好**——升级通道可以整块摘出/接回而不影响主线。

**事实（保留供参考，但定性已改）**：`remote_comfy_upgrade`（root.flow.json:1044-1103）没有入边、没有 `next`，因此当前不进主运行链。`select_delivery_source` 的 merge 里 `comfy_upgrade_bundle` 键当前为空，`_pass_result` 会静默跳过它，`selected_delivery_bundle` 只含 godot 原片——**这正是"关闭升级通道时主链路照常出片"的预期行为**。

**因此这不再计入缺陷清单。** 与之关联的 `upgrade_provider` / `comfyui_url` / `comfyui_workflow_path` / `upgrade_denoise` 输入字段和 `media_style_keyframes` 等 4 个工具，属于"已实现、待接回"的隔离能力，不是"摆设"。

**唯一遗留的真问题**（转由测试台负责）：**静态分析/测试台之前无法把"故意隔离"和"意外断链"区分开**。二者在图上长得一模一样（都是没有入边的节点）。这才是要解决的——见第 15 条：孤立节点需要一个显式标记（如 `params.isolated: true`），让测试台对"标了隔离的"放行、对"没标却断链的"报警。

## 4. 🟠 资产门禁形同虚设：先强制重造 approved 资产，再检查

工位 1 的 fan-out 里，`dispatch_material_reads` 除了连 4 个 read 节点，还连了 `asset_character_forge`（root.flow.json:1284-1287）。

**证据**：`asset_character_forge`（root.flow.json:972-1043）用 `register_mode: approved` 强制锻造 `hero` 和 `night_alley` 两个资产，并写回 `asset_manifest.json`。它的 `next` 是 `plan_asset_check` → `asset_check`。

**问题**：
- 门禁（`asset_check`）的作用本应是"缺料就 `abort_on_failed` 阻断生产"（root.flow.json:337）。但每次运行都先跑 forge **把 hero/night_alley 强行造成 approved 写进 manifest**，门禁再去检查——**永远不可能缺这两个**。门禁退化成了走过场。
- forge 只造 `hero` + `night_alley`，但 `series_bible` 要求的角色是 `hero` + `vendor`（manifest.json:319 默认原料写明"角色：hero、vendor"）。**`vendor` 没有被 forge**，靠 `asset_manifest.json` 里预先登记的 approved 记录（asset_manifest.json:12-18）兜底。

**建议**：门禁节点应在 forge **之前**跑，或把 forge 拆成"仅在缺料时触发"的补料分支（`asset_policy=auto_draft` 时才启用）。当前顺序让"门禁"这个卖点站不住。

## 5. 🔴 被声明 approved 的资产贴图文件在仓库里不存在

**证据**：`asset_manifest.json` 声明 hero/vendor/night_alley 均为 `approved`，并引用：
- `assets/pixel_stage/characters/hero/hero.spritesheet.png`
- `assets/pixel_stage/characters/vendor/vendor.spritesheet.png`
- `assets/pixel_stage/locations/night_alley/layers/*.png`（4 张）

但实际 `assets/pixel_stage/` 目录树里**只有** `godot_project/project.godot` 和 `props/.gitkeep`（见 assets 树列举）。characters/、locations/、drafts/ 目录及其 PNG **全部缺失**。

**后果**：
- 全新 checkout 后，这些 approved 贴图不存在。`hero` 和 `night_alley` 靠运行时 forge 现造还能补上，但 **`vendor` 没有任何节点会造它**（第 4 条），它的 spritesheet 永久缺失。
- **已核实**：`_validate_pixel_assets`（builtin_mcp.py:2718-2790+）**只校验 `profile.json` 是否存在**（`_resolve_pixel_asset_profile` + `resolved.exists()`，:2730-2731 / :2739-2740），以及 profile 里的 `actions` 和 `render_contract` 数值是否达标——**从头到尾不检查 spritesheet PNG / layer PNG 贴图文件是否真实存在**。因此只要 `vendor/profile.json` 在（或被 forge 现造出来），哪怕贴图 PNG 全缺，门禁也返回 `asset_ok=True` 放行。→ 到 `godot_render` 阶段 vendor 角色渲染为空/不可见。这与 `后续搭建计划.md` 末尾提到的"**像素资产渲染修复记录（角色不可见问题）**"高度吻合，说明这个坑已经踩过。这是"资产门禁"卖点的第二处硬伤（第一处见第 4 条）：**它校验的是登记与元数据，不是产物**。
- `.gitignore` 可能忽略了这些生成的 PNG，导致仓库里只有 manifest 不见图——需要确认 forge 是否被当成"每次运行现造"的必经步骤。若是，则 `approved` 这个状态名是误导（它并非"已批准的固化资产"，而是"每次现造")。

## 6. 🔴（用户核心抱怨）MCP 工具库右侧栏无法深入配置：`params_schema` 全空且前端不消费

用户反馈："在 mcp 工具库右侧边栏，点开没有详情，只有机械的、固定参数，无法深入到 mcp 内部。"根因已定位：

**证据链**：
1. manifest 里 15 个 media 工具的 `params_schema` **全部是空对象 `{}`**（manifest.json 各工具，如 :92 :132 :174 …）。
2. 前端 `McpTool` 类型**声明了** `params_schema?: Record<string, any>`（api.ts:172），但 `McpLibraryPanel.tsx` **从头到尾没有读取或渲染它**（全文件搜不到 `params_schema`）。
3. 编辑器只有 6 个死字段：ID / 名称 / Server / Tool / 说明 + 一个 **`默认参数 JSON` 纯文本框**（McpLibraryPanel.tsx:233-239）。用户想配某个工具的参数，只能对着一个 raw JSON textarea 盲填，没有字段名提示、没有类型、没有校验、没有可选项。
4. "快速添加模板"区**只硬编码了 3 个 filesystem 工具**（McpLibraryPanel.tsx:16-47）。这张卡带用的 15 个 media 工具**一个都不在模板里**，无法从 UI 添加，只能手写 manifest。

**这就是"机械、无用的固定参数"的来源**：面板本质是个通用 CRUD，对具体工具的真实参数结构一无所知。

**后果与建议**：
- 每个 media 工具的真实参数（如 `godot_render_pixel_episode` 需要 `shot_plan_path` / `require_godot_native`，`style_keyframes` 需要 `denoise` / `seed` / `workflow_path`）散落在 `builtin_mcp.py` 的 `params.get(...)` 里（如 builtin_mcp.py:266-278），**没有任何地方声明成 schema**，前端自然无从渲染表单。
- 正确做法：让内置工具注册表导出每个工具的参数 schema（名称/类型/默认值/必填/枚举/说明），后端 `/api/lab/flows/{id}/mcp-tools` 把 schema 一起返回，前端 `McpLibraryPanel` 根据 schema 动态渲染字段化表单（而不是一个 JSON textarea）。这样用户才能"深入到 mcp 内部"。
- 短期补丁：至少把 15 个 media 工具也加进 `filesystemPresets` 模板列表，并在编辑器里根据 `default_params` 的键动态生成带 label 的输入行。

## 7. 🟠 硬编码的临时 ComfyUI 隧道地址

**证据**：`https://wheels-certain-hung-believe.trycloudflare.com` 这个 cloudflare 临时隧道 URL 被硬编码在 manifest 至少 4 处（:90 :123 :146 :424）和 root.flow.json（:1084 走的是 `store:episode_brief.comfyui_url` 还好，但 default 仍指向它）。

**问题**：cloudflare quick tunnel 是**临时的、随机域名、重启即失效**。把它当默认值写死，意味着这张卡带交付给任何其他人时，ComfyUI 相关默认配置**必然指向一个已死的地址**。而且这属于把"某人本地的临时外网入口"泄露进了版本库。

**建议**：默认值应回落到 `http://127.0.0.1:8188`（builtin_mcp.py 里的 fallback 本来就是它，见 :160 :272），临时隧道地址交给用户在输入框里填，不进仓库。

## 9. 🟡 fan-out / join 的执行顺序隐患

`dispatch_material_reads` fan-out 到 5 个节点（4 read + 1 forge），它们全部汇入 `plan_asset_check`。

**引擎行为**（root_flow.py:101-139）：`plan_asset_check` 的 `_incoming_counts` = 5，BFS 里用 `completed_parents` 做 join barrier（:112-114），要等 5 个父节点都完成才放行。这一点是**对的**，join 语义正确。

**但隐患在于**：
- `asset_character_forge` 会**每次运行都重写** `asset_manifest.json`（builtin_mcp.py 的 `_record_pixel_*` 系列），而 `load_asset_manifest` 也在同一批 fan-out 里**并发读**同一个文件。当前是单线程 BFS 顺序执行所以侥幸不冲突，但一旦引入并行渲染（`后续搭建计划.md` L4 计划），这里就是竞态：读到的 manifest 可能是 forge 前还是 forge 后，不确定。
- `load_series_bible` 是 `dispatch_material_reads` 的 `next`（唯一"主边"），其余 4 个靠 edges。语义上没问题，但把"哪个是主边"这种隐含约定藏在 `next` vs `edges` 的区别里，可读性差。

## 10. 🟡 `abort_on_failed` 的 strict 判定口径不一致

`_tool_call`（node_executor.py:157-167）里，`strict` 只认 `validation_ok is False` 或 `asset_ok is False` 两个特定键；而 `failed` 又额外看每个 tool result 的 `ok is False`。`abort_on_failed` 只在 `asset_check` 和 `validate_shot_plan` 两个节点开了（root.flow.json:337、617）。

**问题**：`godot_render`、`ffmpeg_mux` 这些真正会因环境缺失（Godot/FFmpeg 不在 PATH）而失败的节点**没开 `abort_on_failed`**。渲染失败后，`lab_failed=True` 会让整个 run 标记 failed（runner.py:248-252），但流程**不会中断**，会继续往下跑 `plan_world_state_update` → `update_world_state`，**在渲染失败的情况下仍然回写 world_state.json**，污染连续性状态。

**建议**：`godot_render`、`ffmpeg_mux` 也应 `abort_on_failed: true`，避免"渲染没成功却把这一集写进世界状态"。

## 11. 🟡 其它零碎

- **`welcome_ui` 的 `next` 与 edges 重复**：state 里有 `next: collect_episode_brief`（root.flow.json:52），edges 里又有同样一条（:1143-1147）。`next_states` 会 dedup（root_flow.py:73-78）所以不出错，但全流程大量这种 `next`+edge 双写，是维护负担、易改漏（正是第 1 条数据链断裂的温床）。
- **`select_delivery_source` 无 `_tool_has_process_parent` 保护但也不需要**：它是 `pass_result`（transfer），不受工具节点必须挂 AI 后的校验（runner.py:187 只查 `action == tool_call`）。但 `remote_comfy_upgrade` 是 `remote_call`，若哪天补上入边，它的父节点得是 process 节点否则……实际 runner.py:187 只拦 `tool_call` 不拦 `remote_call`，所以 remote 节点绕过了这条产品铁律——口径不一致。
- **`shot_count` 输入字段是死的**：manifest.json:331 收集 `shot_count`，但 `_build_pixel_shot_plan`（builtin_mcp.py:564 调用）文档自己承认"内置生成器仍默认 3 镜头"（流程说明.md:100），用户填的镜头数不生效。
- **`_offline_llm_response` 是从 file_summary 卡带抄来的**：它解析 `file_path`/`output_path`/`focus`（node_executor.py:373-375），这些是文件总结卡带的字段，对像素短剧毫无意义，进一步说明兜底逻辑没为这张卡带定制。

## 12. 修复优先级建议

## 13. 🔴（第二轮新增·MCP 系统深度剖析）schema 全链路无人负责，这才是"MCP 很乱"的系统性根因

第一轮定位了前端不消费 `params_schema`。第二轮追到后端与注册表，发现问题是**整条链路没有任何一环生产或校验 schema**：

**证据链（从数据源头到 UI）**：
1. **注册表层**：`BuiltinMcpRegistry`（builtin_mcp.py:30-152）把每个工具注册成一个裸 `callable`，参数全靠函数体内 `params.get("xxx", 默认值)` 就地解析（如 `style_keyframes` 的 denoise/seed/workflow_path 散在 :266-278）。**注册表没有任何地方声明"这个工具接受哪些参数、类型是什么、哪些必填"**。工具的真实参数契约只存在于 Python 函数体里，外部无法内省。
2. **API 层**：`_normalize_mcp_tool`（main.py:485）对 `params_schema` 只做 `raw.get("params_schema") or {}` 的**原样透传**；`create`/`update` 端点（main.py:824-867）把 payload 存进 manifest 就完事，**从不生成、不补全、不校验 schema**。API 完全是个无脑 CRUD。
3. **manifest 层**：15 个 media 工具的 `params_schema` 因此全是 `{}`（第 6 条已列）。
4. **前端层**：`McpLibraryPanel` 不读 `params_schema`，只给一个 raw JSON textarea（第 6 条已列）。

**这就是"乱"的本质**：MCP 工具的参数契约在系统里**没有单一事实来源（single source of truth）**。它同时以三种不一致的形式散落存在：
- Python 函数体里的 `params.get(...)`（真正生效的）
- manifest `mcp_tools[].default_params` 里的示例值（部分工具有，部分没有）
- root.flow.json 各 tool 节点 `tools[].params` 里的硬编码值（真正运行时覆盖的）

用户在右侧栏改 `default_params`，**运行时根本不一定用它**——因为 `_tool_call`（node_executor.py:122-137）优先用节点自带的 `tools[].params`，只有节点没声明 tools 时才回落到库工具的 `default_params`。所以用户在 MCP 库里配的参数，对已经在流程图上绑好的工具节点**多数情况下无效**。这是"配了没用、不知道配的到底是什么"的直接原因。

**系统性修法（按投入从小到大）**：
- **最小**：给每个 media 工具在注册表里补一个 `params_schema` 字面量（name/type/default/required/enum/help），`list_mcp_tools` 时合并返回；前端按 schema 渲染字段化表单。
- **中等**：把"库工具默认参数"与"节点实例参数"的关系在 UI 里显式化——右侧栏编辑的是模板默认值，节点抽屉编辑的是实例覆盖值，两者要能看出继承关系。
- **彻底**：让注册表用带类型注解的 dataclass/pydantic 定义每个工具的入参，schema 由类型自动导出，函数体不再手写 `params.get`。一处定义，API 校验、前端表单、文档全部自动同步。

## 15. 🔴（第三轮核心）测试台太弱：所有严重 bug 都能「测试通过」

这是本轮作者点出的**最关键问题**：上面第 1、4、5 条这些致命 bug，在现有测试台里**全部显示"运行完成 / 通过"**。测试台没能拦住任何一个，反而给用户"卡带没问题"的错觉。这比 bug 本身更危险。

### 15.1 探针只走一条分支（已定位根因）

**前端** `getProbeRange`（TestBenchView.tsx:68-133）用 BFS 求 start→end，但只保留**第一条到达终点的路径**（`path` 是单条数组），一旦命中终点就 `queue.length = 0` 提前退出。分支扇出的其余节点被 `visited` 去重后**永久丢弃**。

**后果**：对本卡带，探针从 `dispatch_material_reads`（fan-out 5 条）到 `plan_asset_check`，**只会走 `load_series_bible` 一条**，另外 3 个 load 节点 + `asset_character_forge` 全部不测。用户以为测了整个资料入库工位，实际只测了 1/5。

**后端** `_normalize_probe_range`（runner.py:296-341）在收到前端 `node_ids` 时直接采信，不做分支补全；`_build_probe_root_flow`（:343-368）还把范围重连成一条**纯线性链**（`state["next"] = node_ids[index+1]`），彻底抹掉分支结构。

**修法**：探针范围应取 start→end 之间**所有路径的节点并集**（含分支），后端保留原始 fan-out/join 边，而不是线性化。

### 15.2 测试台只跑通、不做语义断言（根本缺陷）

即便探针走全了，现有测试台**也发现不了第 1 条数据链断裂**。因为它只判断"节点有没有执行完 / 有没有抛异常"，从不检查：

- 某节点声明的 `input` 键在运行时的 store 里**到底存不存在**（`production_context` 全程为空，但节点照常"完成"）；
- 某个 `output` 键有没有真的被写入；
- 工具节点产出的文件路径是否真实存在（第 5 条贴图缺失）。

`_read_input`（node_executor.py:208-219）对缺失键**静默跳过**，节点依然返回成功。测试台拿到的事件是 `lab_node_executed`，于是标绿。**"跑通"不等于"跑对"，这是测试台当前最大的谎言。**

### 15.3 修复方向（本轮要落地的）

1. **探针分支全覆盖**：前端 `getProbeRange` 改为节点并集；后端保留分支边。
2. **数据链静态检查**：新增一个校验器，遍历 `root.flow.json`，对每个节点的 `input` 键做"生产者可达性分析"——若某 input 键在其所有上游节点的 `output` 里都找不到，报 `dangling_input` 错误。**这条能直接抓出第 1 条。**
3. **孤立节点显式标记**：给故意隔离的节点（如 `remote_comfy_upgrade`）加 `params.isolated: true`；校验器对"标了 isolated 的无入边节点"放行，对"没标却无入边"的报 `orphan_node` 警告。这样才能区分第 3 条（特性）和真正的意外断链。
4. **运行时断言**：测试台运行后，对每个节点核对 `input_value` 是否为空（空且非首节点 → 标黄警告"输入上下文为空"），把 15.2 的静默失败暴露出来。

**验收标准**（作者要求）：先让测试台能**检测出**上面已知的 bug（数据链断裂、孤立 vs 断链），确认测试台真的在干活，**再**去修 bug。
