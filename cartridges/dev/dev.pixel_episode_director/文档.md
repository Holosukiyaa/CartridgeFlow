# Godot 到 ComfyUI 升级通道设计文档

更新时间：2026-07-15

当前状态补充：本文件是升级通道设计稿。`dev.pixel_episode_director` 已按 `CF-FARP@0.1` 完成协议化重写，认证主链暂不接入 ComfyUI 升级通道。当前 `remote_comfy_upgrade` 作为 `params.isolated=true` 的隔离开发支线保留，并以普通 `tool_call` 包装 `media.remote_upgrade_keyframes`，不再作为认证主链的 `remote_call` 能力要求。

本文用于固化 `dev.pixel_episode_director` 后续扩展方向：在现有 Godot / 本地像素片场雏形之上，增加一条独立的 GPU 美术升级通道，把 3080 12GB 的算力接入卡带流程，但不破坏当前已经跑通的工业流水线。

## 1. 核心判断

当前像素导演卡带已经验证了“工业化流程”的基础形态：

```text
入料
-> 资料入库
-> 资产门禁
-> 导演制片
-> 质检
-> 片场渲染
-> 合成出片
-> 连续性回写
```

下一步不应该把 ComfyUI 直接塞进 Godot 渲染节点里，也不应该让 ComfyUI 接管导演职责。

更合理的分工是：

```text
Godot / 本地片场：负责可控性
ComfyUI：负责 GPU 美术增强
卡带流程：负责调度、质检、选择、返工和交付
```

也就是说，Godot 负责稳定产出镜头结构、角色位置、相机调度、基础构图和控制信息；ComfyUI 在这些控制信息之上做风格化、细节增强、局部重绘、放大和关键帧变体。

## 2. 为什么要做升级通道

Godot 到 ComfyUI 的升级通道可以带来几个关键收益：

1. **保留导演控制**

   纯 ComfyUI 容易画面好看但镜头失控。Godot 先确定角色、地点、镜头、遮挡和运动关系，ComfyUI 只做增强，产物更稳定。

2. **让 3080 12GB 用在刀刃上**

   3080 12GB 不适合盲目从零生成长视频，但很适合处理关键帧、局部 inpaint、img2img、ControlNet、upscale 和小批量变体。

3. **降低失败成本**

   如果某个镜头风格跑偏，只重跑该镜头的升级工位，不需要重跑资料入库、镜头表、Godot 基础渲染和连续性回写。

4. **形成卡带技术壁垒**

   壁垒不只是一个 ComfyUI workflow，而是：

   ```text
   shot_plan
   -> Godot control bundle
   -> ComfyUI workflow params
   -> 自动质检 / 人工选择
   -> 局部返工
   -> 合成交付
   ```

## 3. 在现有流程中的位置

建议在当前主链路中新增一段独立工位。早期草案曾考虑把升级段拆成多个工具节点：

```text
godot_render
-> upgrade_channel_prepare
-> comfyui_style_pass
-> upgrade_qc
-> select_delivery_source
-> ffmpeg_mux
```

但实际落地时不采用“工具节点串工具节点”的方式。原因是 ComfyUI 与 FFmpeg、Godot、本地 media helper 不同：它依赖另一台 GPU 机器或网络服务，不能随卡带离线打包带走，所以应该在流程语义上显式标记为 **远程节点**。

当前实现采用：

```text
godot_render
-> remote_comfy_upgrade
-> select_delivery_source
-> ffmpeg_mux
```

其中：

- `godot_render` 仍然输出当前可用的 AVI、帧序列、预览页和 render bundle。
- `remote_comfy_upgrade` 是 `action=remote_call`、`node_category=remote` 的远程节点，代表当前流程依赖一台可访问的 ComfyUI 服务。
- `remote_comfy_upgrade` 内部复用 `media.remote_upgrade_keyframes` 模板，一次完成关键帧提取、ComfyUI 风格化、QC 报告和升级 bundle 汇总。
- 当前第一版 `ffmpeg_mux` 仍然使用 Godot / 本地片场原片出最终视频；`remote_comfy_upgrade` 先作为可质检、可对照、可返工的升级 bundle 产物进入流程。后续再把“通过 QC 的升级关键帧/片段”接入最终合成。

原则：普通 `tool_call` 仍然只代表可打包、可离线或本机可用的工具；不能为了 ComfyUI 放宽所有工具节点的串联规则。

远程节点必须硬失败：如果 ComfyUI 不可达、没有真正使用远程 provider、或 Godot / 本地片场没有输出 PNG 帧序列，流程应停在 `remote_comfy_upgrade`，不能继续静默合成一条与上一版相同的 Godot 原片。

## 4. 升级通道的数据包

升级通道的关键不是直接传一张图，而是把 Godot 产物整理成 ComfyUI 可消费的控制包。

长期目标结构：

```text
render_bundle/
  frames/
    shot_001_frame_000.png
    shot_001_key_start.png
    shot_001_key_end.png
  masks/
    shot_001_hero_mask.png
    shot_001_vendor_mask.png
    shot_001_foreground_mask.png
  maps/
    shot_001_layer_map.png
    shot_001_depth_hint.png
  meta/
    shot_plan.json
    camera.json
    actors.json
    style_prompt.json
    comfy_control_manifest.json
```

第一版最小可用结构：

```text
comfy_control_bundle/
  keyframes/
    shot_001_key.png
    shot_002_key.png
    shot_003_key.png
  meta/
    shot_plan.json
    shot_meta.json
    style_prompt.txt
    comfy_control_manifest.json
```

第一版先不强求 mask、depth、layer map。先跑通关键帧风格化，再逐步加控制条件。

## 5. ComfyUI 外部 MCP 工位

ComfyUI 不直接作为卡带内部代码，而是作为外部 GPU 服务，通过 MCP / media bridge 暴露给卡带。

建议抽象成这些工具：

```text
comfy_ping
comfy_submit_workflow
comfy_get_result
comfy_style_keyframe
comfy_style_keyframe_batch
comfy_upscale_frame
comfy_inpaint_region
comfy_generate_variants
```

第一版只需要实现：

```text
comfy_style_keyframe
```

输入：

```json
{
  "image_path": "test_output/pixel_episode/comfy_control_bundle/keyframes/shot_001_key.png",
  "style_prompt": "pixel 2.5D neon alley, cinematic lighting, clean silhouettes",
  "negative_prompt": "text, watermark, logo, distorted character",
  "workflow_id": "pixel_2_5d_style_v1",
  "seed": 123456,
  "strength": 0.45,
  "output_dir": "test_output/pixel_episode/comfy_upgrade"
}
```

输出：

```json
{
  "ok": true,
  "provider": "comfyui",
  "workflow": "pixel_2_5d_style_v1",
  "source": "test_output/pixel_episode/comfy_control_bundle/keyframes/shot_001_key.png",
  "styled": "test_output/pixel_episode/comfy_upgrade/shot_001_styled.png",
  "seed": 123456,
  "status": "completed"
}
```

## 5.1 当前已实现的基础 media 模板

当前代码层已经先落地一组通用 `media` MCP 模板，作为后续接 ComfyUI / WebUI / FFmpeg / Godot 的基础设备层。

已实现工具：

```text
media/probe
media/extract_keyframes
media/style_keyframes
media/remote_upgrade_keyframes
media/qc_outputs
```

其中：

- `media/probe`：探测 `local`、`ffmpeg`、`comfyui`、`godot` 等 provider 是否可用。
- `media/extract_keyframes`：从 Godot / 本地片场帧序列中提取关键帧，生成 `comfy_control_manifest.json` 和预览页。
- `media/style_keyframes`：对关键帧做美术升级。`provider=local` 时走本地确定性 fallback；`provider=comfyui` 且提供 workflow 后走 ComfyUI API。
- `media/remote_upgrade_keyframes`：远程节点专用的组合模板，内部串起 `extract_keyframes -> style_keyframes -> qc_outputs`，对外输出单个 `remote_media_upgrade_bundle`。
- `media/qc_outputs`：检查升级产物是否存在、图片尺寸是否可读、数量是否满足要求，并输出 QC 报告。

这些工具不是像素短剧专用工具，而是通用 media 工具族的第一批基础模板。像素短剧只是在这些模板上叠加自己的 profile 和参数。流程层用 `remote_call` 标识远端依赖，工具层用 `media.*` 复用文件处理、上传、下载和 QC 逻辑。

## 6. 升级结果契约

ComfyUI 工位不能只吐图片，必须吐结构化结果，便于后续质检和选择。

建议输出：

```json
{
  "ok": true,
  "provider": "comfyui",
  "workflow": "pixel_2_5d_style_v1",
  "source_bundle": "test_output/pixel_episode/comfy_control_bundle",
  "outputs": [
    {
      "shot_id": "shot_001",
      "base": "test_output/pixel_episode/comfy_control_bundle/keyframes/shot_001_key.png",
      "styled": "test_output/pixel_episode/comfy_upgrade/shot_001_styled.png",
      "seed": 123456,
      "prompt": "pixel 2.5D neon alley...",
      "status": "completed"
    }
  ],
  "preview_html": "test_output/pixel_episode/comfy_upgrade.preview.html",
  "manifest": "test_output/pixel_episode/comfy_upgrade_manifest.json"
}
```

后续 `upgrade_qc` 节点只读这个 manifest，不直接猜文件路径。

## 7. 质检和返工

升级通道必须有门禁，不能默认 ComfyUI 输出就是最终产物。

第一版质检可以很简单：

- 文件是否存在
- 输出尺寸是否符合要求
- 每个 shot 是否都有对应 styled keyframe
- 是否生成 preview HTML
- 是否保留原始 shot_id 映射

后续再增加：

- 角色轮廓是否偏离过大
- 主要颜色是否符合风格
- 构图是否保留
- 多镜头风格是否统一
- 人工选择 preferred variant

质检结果建议：

```json
{
  "ok": true,
  "passed": true,
  "issues": [],
  "selected_outputs": [
    {
      "shot_id": "shot_001",
      "image": "test_output/pixel_episode/comfy_upgrade/shot_001_styled.png"
    }
  ],
  "fallback": "godot_render_bundle"
}
```

如果升级失败，流程不应该中断整片生产。第一版策略：

```text
ComfyUI 通过 -> ffmpeg_mux 使用升级产物或升级预览
ComfyUI 失败 -> ffmpeg_mux 使用 Godot 原始 render_bundle
```

## 8. 分阶段落地路线

### 阶段 0：保留当前主链路

目标：不破坏现有可运行能力。

- `godot_render -> ffmpeg_mux` 继续可用。
- `episode.mp4` 继续能生成。
- `world_state.json` 继续能回写。

### 阶段 1：关键帧升级

目标：打通 Godot 到 ComfyUI 的最短路径。

新增节点：

```text
extract_godot_keyframes
comfy_style_keyframes
upgrade_qc
```

只处理每个 shot 的一个关键帧，不碰全视频。

验收：

- 能从 Godot 帧序列中提取 3 张 keyframe。
- 能生成 3 张 styled keyframe。
- 能生成对照预览 HTML。
- 失败时可以回退到 Godot 原始输出。

### 阶段 2：每镜头首尾帧

目标：开始关注镜头内一致性。

- 每个 shot 提取 start / end 两张关键帧。
- ComfyUI 输出对应 styled start / styled end。
- 预览页展示原始帧和升级帧对照。

### 阶段 3：短序列增强

目标：从静态关键帧走向小段镜头升级。

- 每个 shot 抽样若干帧。
- ComfyUI 批处理风格化。
- 允许只升级重点镜头。

### 阶段 4：全片升级

目标：在质量和速度可控后，再考虑全片增强。

- 全帧批处理
- 插帧
- 超分
- 去闪烁
- 统一色彩

这一阶段不要提前做，避免一开始就被显存、速度和一致性问题拖住。

## 9. 对现有卡带的修改建议

短期建议只加 3 个节点：

```text
extract_godot_keyframes
comfy_style_keyframes
upgrade_qc
```

并在 `ffmpeg_mux` 前增加一个选择节点：

```text
select_delivery_source
```

选择逻辑：

```text
if upgrade_qc.passed:
    use comfy_upgrade output
else:
    use godot render_bundle
```

中期再把升级通道拆成独立小卡带：

```text
dev.comfy_keyframe_upgrade
```

母卡带只负责传入 Godot control bundle，等待升级卡带返回 manifest。

## 10. 当前不做的事情

为了尽快落地，当前先不做：

- Blender 片场
- Unreal 片场
- 全视频文生视频
- 长序列视频扩散
- 复杂角色骨骼重绘
- 多角色一致性大模型训练

当前目标只有一个：

```text
让 Godot 产物稳定变成 ComfyUI 可消费的控制包，并跑通第一版关键帧美术升级。
```

## 11. 一句话总结

Godot 是导演和片场，ComfyUI 是 GPU 美术工位，卡带是工业调度系统。升级通道的价值是把三者连接成可复现、可质检、可返工、可逐步升级的生产线。
