# CartridgeFlow Roadmap

> 这份文档只记录会改变产品能力边界的阶段目标，不记录日常修改和完成历史。
> 当前协议基线：`CARTRIDGEFLOW-BASE@0.2`、`CF-FARP@0.6`
> 更新日期：2026-07-21

## 1. 产品目标

CartridgeFlow 要把专业 AI 工作流交付为一张完整的应用卡带：它可以安装、预检、运行、升级、回滚和卸载，并拥有自己的流程、界面、工具配方、资源声明、人工决策点和交付标准。

底座提供通用运行能力，不拥有任何具体行业业务。开发者负责制作卡带，最终用户只使用卡带提供的专业界面，不必理解流程图、模型参数或外部工具内部结构。

```text
卡带 = 产品契约 + Root Flow + 私有 DLC + 专属 UI
     + 模型与工具配方 + 资源声明 + 质量门槛
     + 测试、迁移、恢复和卸载策略
```

## 2. 所有权边界

| 层级 | 应当拥有 | 不应拥有 |
| --- | --- | --- |
| 底座核心 | 协议解释、运行状态、交互、恢复、产物、依赖预检和 DLC 生命周期 | 卡带 ID、领域提示词、领域数据结构和专用 UI 分支 |
| 通用适配层 | 模型、MCP、HTTP、CLI、文件和媒体的通用连接能力 | 固定业务工作流、具体模型配方和领域默认值 |
| 卡带 | Root Flow、专属前后端、领域协议、配方、资产、测试和迁移 | 修改全局注册表或读取其他卡带私有状态 |
| 本地配置 | URL、密钥、可执行文件路径和用户自己的资源绑定 | 跟随卡带传播的凭据和机器私有信息 |
| 用户数据 | 输入、工程文件、运行记录、产物和导出包 | 在普通卸载时被删除 |

判断标准很简单：删除某张卡带后仍对所有卡带有意义的能力，才可能属于底座；应随该卡带消失的能力，必须由卡带自己拥有。

## 3. 明确不做

- 不把底座做成通用 Prompt、RAG、Agent 或模板市场。
- 不在底座中内置视频、3D、动画等具体业务卡带。
- 不重写 ComfyUI、Blender、Dify 等成熟工具的内部能力，只管理它们的连接、权限、调用和失败恢复。
- 不用 mock、占位产物或静默 fallback 冒充正式运行成功。
- 不在可靠性、安装生命周期和交付证据尚未闭环时优先建设市场、排行等外围能力。
- 不要求最终用户进入流程编辑器或诊断台才能完成专业任务。

## 4. 路线图收录规则

只有满足下列至少一项的工作才进入本路线图：

- 为开发者或最终用户提供一项完整的新能力。
- 改变跨模块契约、卡带生命周期或底座与卡带的责任边界。
- 直接决定生产发布、安全、数据完整性或故障恢复是否可信。

局部样式、文件改名、单个接口修补、补一条测试和普通重构不进入路线图。它们应作为实现过程、Issue 或提交记录存在。

## 5. 阶段里程碑

### 阶段 A：可信的通用底座（当前）

目标：底座声明的每项能力都有真实实现和自动证据，卡带配方能够安全解析为本机资源。

完成标准：

- `BASE@0.2` 与 `FARP@0.6` 的条款、实现和测试可以相互追溯。
- 没有证据的能力明确标记为部分支持，不用文档承诺代替实现。
- 模型、MCP、HTTP 和 CLI 等本地资源通过角色绑定供卡带使用，凭据不进入卡带。
- 错误、重试、检查点、回滚和人工恢复使用统一且可观察的运行语义。
- 干净底座不携带任何业务卡带、领域工具或领域界面。

### 阶段 B：完整的卡带生命周期

目标：卡带从一个可复制目录变成可安全分发和维护的应用单元。

完成标准：

- 卡带包具有清单、文件校验、依赖锁定、权限声明和兼容范围。
- 预检在执行卡带代码或产生业务副作用前完成。
- 安装和升级是原子的，任一步失败都能恢复到旧版本或一致的未安装状态。
- 停用和卸载会移除卡带能力、进程、路由和私有状态，但默认保留用户产物。
- 每次生命周期操作都能生成可检查的报告。

### 阶段 C：卡带专属产品界面与交付闭环

目标：卡带可以拥有真正的专业工作台，运行结果可以被证明、修改和交付。

完成标准：

- 底座提供版本化、中立且受权限约束的 DLC UI Host API。
- 两张结构完全不同的卡带界面可以同时运行，样式、脚本、状态和数据互不影响。
- 每个正式产物都有来源、输入、工具、版本、审批和 revision 记录。
- 上游修改会使受影响的下游产物失效，旧版本保留但不会冒充最新交付。
- 最终用户能在卡带界面完成输入、生成、修改、审批和导出。

### 阶段 D：真实卡带验收与可信分发

目标：用一张独立于底座的真实卡带证明整个体系可以交付，并形成可复用的开发和发布工具链。

完成标准：

- 参考卡带位于独立仓库或独立发布包，底座仍保持零业务卡带。
- 参考卡带按公开支持矩阵完成安装、预检、运行、人工修改、恢复、交付、升级和卸载。
- 真实外部依赖失败会映射为稳定错误和可执行恢复动作，不依赖服务端日志才能处理。
- 开发者可以用正式 schema、SDK 和 CLI 创建、校验、测试、打包和预检卡带。
- 分发包具备签名、完整性校验、权限差异提示和兼容性判断。

## 6. 生产发布门槛

CartridgeFlow 只有同时满足以下条件，才应被称为可用于生产的卡带底座：

- 所有声明支持的协议能力都有自动化证据，测试结果可重复生成。
- 正式底座和发布包中没有业务卡带、密钥、用户数据、日志、缓存或测试产物。
- 卡带安装和升级失败不会破坏已有可用版本。
- 普通卸载不留下功能残留，也不会删除用户产物。
- 可恢复故障经过进程重启、外部工具失败和副作用重放测试。
- 最终产物具有完整来源，缺失或失效的主产物不能被标记为成功交付。
- 最终用户不打开流程图也能完成支持范围内的主任务。

## 7. 决策原则

1. 先证明底座能力，再扩充外围生态。
2. 先定义所有权和失败语义，再添加实现。
3. 配方随卡带迁移，凭据和机器路径留在本地。
4. 正式运行宁可明确失败，也不生成看似成功的替代结果。
5. 新协议版本是完整、只读、自包含的快照；旧协议支持范围由明确策略决定，不在实现中无限累积。
