# 数字替身与 ComfyUI 互补生产行动指南

更新日期：2026-07-18
适用范围：`dev.series_3d_episode_factory` 以及后续系列视频卡带

本文件是从“真实 3D 资产 -> Blender 单镜头 -> ComfyUI 创作重绘”继续推进时的执行规范。它补充 `GOAL.md`、`TODO.md` 和 `ENVIRONMENT_BASELINE.md`，不修改 CartridgeFlow 协议语义。

## 一、核心决定

数字替身化和 ComfyUI 互补不是两个互斥方向，而是一条生产链的两端：

```text
用户文本与审美决定
        |
        v
CartridgeFlow / LLM：拆镜头、选资产、编译约束、记录失败
        |
        v
Blender 数字替身：动作、脚底接触、空间、摄影机、时间轴
        |
        v
Shot Control Bundle：姿态、深度、遮罩、边缘、参考图、参数
        |
        v
ComfyUI：角色替换、场景重绘、材质、灯光、统一风格
        |
        v
人工复核：确认、驳回、带失败经验局部重跑
        |
        v
MP4 交付，同时保留 Blender 原片作为兜底
```

Blender 不是最终美术渲染器，而是确定性摄影棚。当前男性模型应被视为“动作替身”，它的价值是稳定骨骼、动作、位置和镜头关系，不是把它的肌肉、服装或脸保留到成片中。

ComfyUI 不是生产调度器，也不保存系列世界状态。它负责在已经确定的镜头约束内创造角色外观、场景细节、灯光、材质和风格。

## 二、五条不可违反的原则

### 1. 确定性和创造性必须分层

下列内容默认锁定：

- 帧率、帧数和镜头时长。
- 摄影机位置、运动方向和基本构图。
- 人物的骨骼运动、行走方向和脚底接触。
- 角色与地面、墙体、道具的遮挡关系。
- 用户明确标记为“不可改变”的道具或资产。

下列内容可以交给 ComfyUI 创造：

- 角色种族、服装、发型、材质和画风。
- 房屋、道路、植物、天空和远景细节。
- 灯光色温、阴影风格和材质质感。
- 不影响叙事的布景细节。

如果一个内容同时属于两类，必须通过遮罩或分层明确优先级，不能靠提示词猜测。

### 2. 相同提示词不是稳定性的充分条件

系列稳定性由以下组合决定：

```text
模型文件与哈希
+ Workflow 版本
+ 角色参考图组
+ 世界参考图组
+ Style Pack
+ 控制视频与遮罩
+ seed 和采样参数
+ 上一镜头的连续性锚点
```

LLM 可以编译和维护这些内容，但不能只输出一段自由文本提示词就声称角色已经锁定。

### 3. 每个镜头都保留原始回退

ComfyUI 输出必须与 Blender 原片并列保存。ComfyUI 失败、显存不足、身份漂移或用户不喜欢风格时，系统应回退到 Blender 预演，而不是生成一个未标记的假成功结果。

### 4. 先做单镜头纵向闭环，再扩展镜头数量

在一个 2 至 3 秒镜头上没有通过动作、身份、连续性和资源质量门之前，不扩展到整条 15 至 30 秒样片，也不同时接入第二套视频工作流。

### 5. 角色替换优先使用姿态和遮罩，而不是人物边缘

全画面 Canny 会把当前替身的体型轮廓传递到输出。它适合控制房屋和道路结构，却会阻碍动漫角色拥有不同身材。角色替换模式应优先使用骨骼姿态、深度和动态人物遮罩；Canny 仅用于背景或低强度辅助。

## 三、职责边界

| 层 | 负责内容 | 不负责内容 |
| --- | --- | --- |
| 用户 | 角色、场景、道具、风格和许可来源的最终选择 | 手工维护 ComfyUI 节点连接 |
| LLM / CartridgeFlow | 剧本拆解、镜头规划、资产匹配、提示词编译、工作流选择、失败分类和回滚 | 直接替代 Blender 的空间事实，或伪造生成结果 |
| Blender | 真实资产、骨骼动作、走位、摄影机、时间轴、深度、姿态、遮罩和预演 | 最终承担所有风格化美术 |
| ComfyUI | 角色重绘、场景重绘、材质、灯光、风格和短视频生成 | 保存跨镜头世界状态、决定叙事或隐藏失败 |
| 人工复核 | 判断故事、角色、风格和穿帮是否可接受 | 逐帧手工修补整条视频 |

## 四、数字替身轨道

### A0：冻结动作替身

当前 65 根骨骼的男性角色和 `Walk_Loop` 作为第一条校准镜头的动作替身。此阶段不继续优化他的最终外观，只确认：

- 面朝方向和世界位移一致。
- 脚底接触地面，没有滑步或悬空。
- 摄影机运动和人物屏幕空间轨迹稳定。
- 首帧、末帧和中间帧可被 ComfyUI 读取。

### A1：输出 Shot Control Bundle

每个镜头必须生成同帧率、同帧数、同宽高的控制包。第一版包含：

| 文件 | 作用 | 第一版要求 |
| --- | --- | --- |
| `preview.mp4` | Blender 可观看预演和最终回退 | 必须可播放 |
| `reference.png` | 镜头参考帧或角色参考合成图 | 记录来源和用途 |
| `character_mask.mp4` | 标记人物区域 | 白色为生成/替换区域，黑色为保留区域 |
| `depth.mp4` | 控制空间深度、遮挡和透视 | 不能因材质颜色改变空间顺序 |
| `pose.mp4` 或姿态数据 | 控制骨骼动作 | 不携带原人物的肌肉轮廓 |
| `background_edges.mp4` | 控制建筑、道路和镜头结构 | 只在场景重绘模式使用 |
| `shot.json` | 生产元数据和参数 | 必须可复现 |

VACE 的遮罩约定必须在转换时固定：需要模型重新生成的区域为白色，需要保留控制输入的区域为黑色。所有遮罩都要在 manifest 中写明，不允许每个节点各自解释。

### A2：`shot.json` 最小字段

```json
{
  "schema": "cartridgeflow.shot_control_bundle.v1",
  "shot_id": "pilot_01_walk_01",
  "fps": 12,
  "frame_count": 33,
  "width": 480,
  "height": 832,
  "character_id": "surrogate_male_01",
  "world_id": "suburban_daylight_01",
  "action_id": "walk_forward",
  "camera_id": "side_tracking_01",
  "control_bundle_version": 1,
  "source_preview": "preview.mp4",
  "controls": {
    "character_mask": "character_mask.mp4",
    "depth": "depth.mp4",
    "pose": "pose.mp4",
    "background_edges": "background_edges.mp4"
  },
  "seed": 3946
}
```

实际项目可以增加路径、许可、镜头焦段和提示词字段，但不能删除上述可复现字段而只保留自然语言说明。

### A3：数字替身验收

数字替身轨道通过的条件：

1. 控制视频帧数、帧率和 manifest 完全一致。
2. 人物屏幕空间运动方向与 Blender 预演一致。
3. 左右脚接触时刻稳定，不能出现连续滑步。
4. 遮罩不漏手、脚和头部，也不能大面积吞掉背景。
5. 深度中人物、地面、建筑的前后关系正确。
6. ComfyUI 不需要重新猜测摄影机和动作语义。

代理的皮肤、服装、面部和材质不属于这一阶段的验收项。

## 五、ComfyUI 互补轨道

### B0：当前 3080 基线

第一阶段只使用已经在本机实测的 Wan2.1 VACE 1.3B：

- 运行目标：`480x832`，17 帧短测，33 帧镜头测。
- 单次只运行一个任务，不并行抢占 12GB 显存。
- 统一记录 workflow、模型、seed、步数、耗时和驱动峰值显存。
- 先用 1.3B 证明控制和角色替换逻辑，再评估 14B、量化或更高显存硬件。

本机环境和当前 Canny 结果见 `ENVIRONMENT_BASELINE.md`。当前全画面 Canny 版本适合场景重绘，不应直接当作最终人物替换版本。

### B1：三组隔离实验

所有实验使用同一镜头、同一帧数、同一 seed 和同一输出分辨率。

#### 实验 1：只换场景

- 人物区域尽量保留。
- 背景使用 Canny 或 Depth 控制。
- 目标：确认房屋、道路、灯光和材质可以被重绘成更讨喜且连续的世界。

#### 实验 2：只换人物

- 背景使用保留或低强度控制。
- 人物区域使用动态遮罩。
- 输入原创或已授权的动漫角色全身参考图。
- 人物控制优先使用 Pose、Depth 和遮罩，不把原男性轮廓作为强约束。
- 目标：确认同一段走路动作可以驱动完全不同的角色。

#### 实验 3：人物和场景一起换

- 只锁定动作、摄影机、镜头时长和主要空间结构。
- 人物参考图与世界参考图同时输入。
- 目标：验证“文本决定风格和世界，Blender 决定运动和摄影”的完整模式。

### B2：三种用户模式

卡带最终不向用户暴露 ComfyUI 节点，而暴露三种意图：

| 模式 | 默认保留 | 默认允许改变 | 用途 |
| --- | --- | --- | --- |
| 稳妥模式 | 场景结构、人物动作和摄影机 | 材质、灯光、细节 | 连续剧情、关键道具 |
| 角色替换模式 | 背景、摄影机和动作 | 人物身份、服装、发型 | 固定替身驱动不同演员 |
| 创作模式 | 动作、镜头和叙事构图 | 人物、场景、灯光和材质 | 风格化远景、世界重绘 |

每次生成前，用户应能看到“保留什么、允许改变什么”和参考资产摘要。

### B3：后续 14B 路线

ComfyUI 本机已带有 SCAIL-2 角色替换蓝图和 `WanSCAILToVideo` 节点，但 14B FP16 不是当前 3080 12GB 的第一工作流。只有在 1.3B 控制包和验收标准明确后，才进行以下一种升级：

- 安装适配的量化模型并重新测显存。
- 增加系统内存和 offload，接受更长耗时。
- 使用更大显存设备进行 SCAIL-2 或 Wan Animate 基准。

升级模型不应改变 `Shot Control Bundle` 接口。模型换了，输入契约、输出报告和质量门仍然相同。

## 六、资产包和 LLM 契约

### 1. `CastPack`

角色包至少包含：

- 角色唯一 ID。
- 一张全身主参考图。
- 正面、三分之四、侧面或背面参考图（可逐步补充）。
- 固定服装、发型、颜色和不可改变特征。
- 许可来源或原创声明。
- 角色正向提示词和负向提示词。

为了降低 IP 风险，第一轮替换实验优先使用原创动漫角色，不直接使用未授权的现有商业角色。

### 2. `WorldPack`

世界包至少包含：

- 世界唯一 ID。
- 建筑、道路、植物、天空和光照描述。
- 一张或多张风格参考图。
- 允许变化的布景元素。
- 必须连续的地标和道具。

### 3. `StylePack`

风格包至少包含：

- 模型文件和哈希。
- ComfyUI workflow 版本。
- 正负提示词模板。
- 分辨率、帧数、步数、采样器、scheduler 和 seed 规则。
- 色板、光照和材质要求。
- 已知失败模式。

### 4. LLM 输入与输出

LLM 接收：

- 用户短文本。
- CastPack、WorldPack、StylePack。
- 镜头目标和用户标记的不可变区域。
- 上一次结果和失败记录。

LLM 输出结构化 `ShotSpec`，至少包括：

- 镜头类型和时长。
- 使用哪个动作、摄影机和控制包。
- 角色、世界和风格引用。
- 保留区域与可重绘区域。
- ComfyUI 工作流和参数覆盖。
- 正向提示词、负向提示词和连续性检查项。
- 失败时的下一条回滚路径。

LLM 可以提出参数变体，但不能在没有用户确认的情况下静默扩大生成范围、切换模型或覆盖用户资产。

## 七、失败分类和回滚

每次驳回必须记录一个或多个标准标签：

| 标签 | 典型原因 | 下一步动作 |
| --- | --- | --- |
| `motion_reversed` | 前进变后退、镜头方向错误 | 回到 Blender 控制包，检查位移和姿态方向 |
| `surrogate_leak` | 原男性体型、服装或脸残留 | 降低人物边缘控制，扩大遮罩，改用 Pose/Depth |
| `identity_drift` | 动漫角色脸、发型或服装不稳定 | 增加角色参考图，锁定 CastPack，减少自由度 |
| `scene_drift` | 房屋、道路或地标在帧间变化 | 加强背景 Depth/Canny，缩短分段并固定世界参考 |
| `temporal_flicker` | 颜色、纹理或灯光跳变 | 降低风格自由度，固定 seed，使用相邻片段延续 |
| `contact_error` | 脚滑、悬空、阴影错误 | 不在 ComfyUI 中修动作，回 Blender 修正控制包 |
| `style_mismatch` | 画风或色板不符合 StylePack | 只修改风格包或参考图，不改动作控制 |
| `resource_limit` | OOM、超时或并发冲突 | 降低分辨率/帧数，单任务运行，记录真实失败 |

回滚必须回到产生问题的最近节点，不默认重跑整条流程。每次重跑携带上一次失败标签和用户意见，形成可审计的失败经验。

## 八、质量门

### 技术门

- ComfyUI 返回成功，输出文件可播放。
- 分辨率、fps、帧数和时长符合 `ShotSpec`。
- 任务无未记录的显存溢出、断帧或静默降级。
- 原始 Blender 片和增强片均已保存。

### 动作门

- 人物屏幕空间位移方向正确。
- 关键脚步时刻与控制包一致。
- 摄影机运动和构图没有突然跳变。

### 角色门

- 角色身份在首帧、中段和末帧可辨认。
- 服装、发型、颜色和体型没有无解释的变化。
- 替身原外观没有残留到不应保留的区域。

### 连续性门

- 背景地标、道路边缘和主要道具没有逐帧生长或消失。
- 灯光方向和色温在镜头内保持一致。
- 片段拼接处能使用上一片段的末帧或延续条件。

### 创作门

- 场景和人物确实比 Blender 预演更符合 StylePack。
- 变化是可解释、可重复、可由用户控制的，而不是随机穿帮。
- 用户知道哪些内容被保留、哪些内容被 ComfyUI 重新创作。

## 十、阶段完成定义

### 控制包完成

一个真实 Blender 镜头可以稳定生成完整控制包；所有文件帧数一致；预演方向、脚步、遮罩和深度通过检查。

### 角色替换完成

同一动作镜头可以由至少一个新角色参考图稳定驱动，人物身份在首帧、中段和末帧保持，且没有明显原替身残留。

### 创作重绘完成

同一镜头可以在稳妥模式和创作模式下产生两种可解释结果；用户能看出保留边界；失败可以按标签局部回滚。

### 生产链完成

从文本、资产包、控制包到 MP4 的每一步都有 manifest、workflow、日志、复核点和原始回退。没有通过这一条链路前，不宣称“任意文本一键成片”。

## 十一、明确不做

- 不把当前男性替身的外观当作系列永久角色。
- 不用提示词代替姿态、深度和遮罩控制。
- 不在 3080 12GB 上同时运行多个视频任务。
- 不为了追求更高模型参数而跳过 1.3B 的控制与验收闭环。
- 不把 ComfyUI 生成的随机变化伪装成“稳定风格”。
- 不在首个角色替换镜头通过前扩展到整季或长视频。
- 不直接使用未授权的商业动漫角色作为公开发布素材。

这份行动指南的下一项实际产物不是更多提示词，而是第一个可复用的 `Shot Control Bundle` 和一组“场景替换 / 人物替换 / 全部替换”的对照视频。
